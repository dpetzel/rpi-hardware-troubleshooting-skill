# Pi 3B/3B+ LAN9514 USB/Ethernet Hub

## Overview

The **LAN9514** (Microchip/SMSC) is a combined USB 2.0 hub + 10/100 Ethernet controller used on the Pi 3B, 3B+, and 2B. It connects to the SoC via a single internal USB 2.0 port and provides:

- 4 downstream USB 2.0 ports
- 10/100 Ethernet (3B) or 10/100/1000 Ethernet (3B+ uses LAN7515 variant)

**Critical implication**: All USB ports AND Ethernet share a single upstream USB 2.0 connection to the SoC. If the LAN9514 fails, **all USB and Ethernet die simultaneously**.

## Identifying the Chip

- **Pi 3B**: LAN9514 (QFN-64 package, ~9mm × 9mm, top side of board near USB/Ethernet connectors)
- **Pi 3B+**: LAN7515 (similar function, adds Gigabit Ethernet PHY, but Gigabit is bottlenecked to ~300Mbps by USB 2.0 upstream)
- Labeled "SMSC" or "Microchip" on the package

## Failure Symptoms

### All USB + Ethernet Dead

**Observation**: No USB devices enumerate, no Ethernet link LEDs, but board boots (green LED blinks, HDMI output works).

**Verification**:
- `lsusb` shows only the root hub (no `0424:9514` device)
- `dmesg` shows USB hub enumeration failure or timeout
- Ethernet LEDs (PP22, PP23) never illuminate

**Cause**: LAN9514 chip failure — either dead or not receiving power/clock.

### LAN9514 Shorting the 3.3V Rail

**Observation**: Board appears dead or partially dead. Red LED on but no boot activity.

**Diagnosis**:
1. Measure PP8 (3.3V rail) — if reading well below 3.3V (e.g., 0.5–1.5V), something is dragging the rail down
2. Power off, measure resistance from PP8 to GND — if very low (<5Ω), there's a short
3. Power on briefly and feel for heat — a failed LAN9514 often gets noticeably hot
4. Advanced: inject current into 3.3V rail with bench supply (current-limited to 500mA) and thermal-image or feel for the heat source

**Common scenario from forums**: PP8 reads ~0.8V instead of 3.3V. Injecting power into the 3.3V rail reveals LAN9514 getting hot. The chip has an internal short and is sinking all available current.

### Intermittent USB Disconnections

**Observation**: USB devices randomly disconnect/reconnect, `dmesg` shows disconnect spam.

**Causes** (in order of likelihood):
1. **Insufficient power supply** — most common. The LAN9514 and downstream USB devices need adequate current. Try a better PSU first.
2. **EMI/interference** — external noise coupling into USB lines
3. **Failing chip** — early stage of LAN9514 degradation

**Diagnostic**: `dmesg | grep -i "usb\|disconnect\|overcurrent"` — look for overcurrent warnings vs simple disconnects.

### Ethernet Only Dead (USB Works)

**Observation**: USB ports work fine, but no Ethernet link.

**Causes**:
- Ethernet magnetics/transformer failure (between LAN chip and RJ45 jack)
- Physical damage to RJ45 connector
- Specific Ethernet PHY section of LAN chip failed (rare)

**Diagnostic**:
- Check Ethernet LEDs at PP22 (activity) and PP23 (link)
- Try different cable and switch port
- `ethtool eth0` — does the interface exist but show "Link detected: no"?

## USB Polyfuses (Pi 3B Only)

The Pi 3B has **individual polyfuses** for USB port pairs (providing overcurrent protection per port). The Pi 3B+ removed these for cost/space.

**Pi 3B**: If a single USB port pair is dead but others work:
- The polyfuse for that pair may be tripped or damaged
- Measure voltage at the USB port's VCC pin — should be ~5V
- Compare to PP27 (USB VCC upstream)

**Pi 3B+**: No individual port fuses — all USB power comes from a single source. If one port has no power, they all should be dead (or it's a connector issue).

## LAN9514 Power Requirements

The LAN9514 needs:
- 3.3V main power (from the board's 3.3V rail)
- 1.8V core (internally regulated from 3.3V on some variants, or from board 1.8V rail)
- 25MHz crystal oscillator (next to the chip on PCB)

If the crystal is damaged or its load capacitors are missing, the chip won't initialize even with correct power.

## Repair Considerations

**Is it worth replacing?**

Generally no, unless you're doing it for learning. The LAN9514/LAN7515 is a BGA/QFN package requiring precise rework. A replacement Pi 3B costs very little compared to the effort.

**Workarounds for a dead LAN9514:**
- Use a USB WiFi dongle (won't work if the entire USB subsystem is dead via the chip)
- If only Ethernet is dead: use WiFi (built into Pi 3B/3B+, independent of LAN9514)
- If entire USB is dead: the board is essentially limited to GPIO/CSI/DSI/HDMI use only — may still be useful for headless projects using WiFi + serial console

## Diagnostic Flowchart

```
All USB + Ethernet dead?
├── Yes → Check PP8 (3.3V rail)
│   ├── PP8 < 2V → LAN9514 likely shorting rail (feel for heat)
│   │   └── Confirm: power off, measure PP8-to-GND resistance
│   ├── PP8 = 3.3V → LAN9514 not receiving clock or has internal failure
│   │   └── Check crystal oscillator, check `lsusb` output
│   └── PP8 = 0V → Regulator dead (see pi3_power_management.md)
├── Only Ethernet dead → Physical layer (cable/connector/magnetics)
└── Intermittent → Power supply first, then EMI, then chip degradation
```
